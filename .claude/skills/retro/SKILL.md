# 会话复盘 Skill

## 你的任务

回顾当前会话的完整对话历史，执行一次结构化复盘，并将有价值的经验持久化。

## 复盘流程

### Step 1: 分析当前会话

回顾对话，识别以下类别的事件：

- **BUG/错误**: 遇到的 bug、报错、测试失败及其根因
- **架构决策**: 做出的设计选择，哪些有效、哪些需要回退
- **效率问题**: 走弯路的地方、重复劳动、本可避免的返工
- **知识盲区**: 对框架/库/API 的误解或不熟悉
- **模式发现**: 发现的代码模式、反模式、或项目特有的约定

### Step 2: 提取可操作的教训

对每个识别到的事件，提炼为：
- **一句话规则**（祈使句，如"数据库迁移前先备份"）
- **原因**（为什么这很重要，1 句话）
- **分类标签**

只提取**通用可复用**的教训。以下内容不应提取：
- 一次性的、不会再遇到的特殊问题
- 显而易见的常识（如"写代码前先理解需求"）
- 与具体业务逻辑绑定、无法泛化的经验

### Step 3: 读取现有教训文件

读取 `.claude/rules/lessons-learned.md` 文件（如果存在）。

### Step 4: 去重与合并

对比新教训和已有教训：
- 如果已有相同或高度相似的教训 → 跳过
- 如果已有相关教训但新发现补充了细节 → 更新已有条目
- 如果是全新教训 → 追加到对应分类下

### Step 5: 写入文件

将合并后的结果写入 `.claude/rules/lessons-learned.md`，严格遵循以下格式：

```markdown
# Lessons Learned

> 从开发会话中提炼的经验教训。此文件由 /retro 自动维护，每条规则在所有未来会话中生效。

## Code Quality
- **[规则]**: 原因 <!-- date:YYYY-MM-DD -->

## Architecture
- **[规则]**: 原因 <!-- date:YYYY-MM-DD -->

## Testing
- **[规则]**: 原因 <!-- date:YYYY-MM-DD -->

## Performance
- **[规则]**: 原因 <!-- date:YYYY-MM-DD -->

## Security
- **[规则]**: 原因 <!-- date:YYYY-MM-DD -->

## Workflow
- **[规则]**: 原因 <!-- date:YYYY-MM-DD -->

## Project-Specific
- **[规则]**: 原因 <!-- date:YYYY-MM-DD -->
```

### Step 6: 输出复盘摘要

向用户展示本次复盘结果：
- 本次会话发现了几个问题
- 新增了几条教训（列出）
- 更新了几条已有教训（列出变更）
- 跳过了几条（与已有重复）
- 当前教训总数

## 重要约束

- 每个分类下最多保留 15 条规则。超出时移除最旧且最少触发的条目
- 规则必须是祈使句，简洁明了，可直接作为编码指令执行
- 不要写空的分类。如果某个分类没有内容，省略该分类标题
- 日期注释用 HTML comment 格式，不影响阅读
